## 推理模型
- **非结构化推理（Non-Structured Reasoning）**：无固定步骤，依赖直觉、经验或零散信息拼接结论，推理过程无明确脉络，类似人类 “灵光一闪” 的思考模式。
   ```markdown
   场景：回答 “周末去公园还是商场”
   
   Prompt：
   “帮我决定周末去公园还是商场，说说你的想法就行，不用分步骤。”
   ```

- **思维链 Chain-of-Thought）**：通过引导模型逐步推理来提升复杂问题解决能力。
   ```markdown
   场景：解数学题 “一个长方形周长 30cm，长比宽多 3cm，求面积”

   Prompt：
   “请一步步解这道题，说清楚每一步的思路：
   1. 先回忆长方形周长公式；
   2. 设未知数表示长和宽；
   3. 根据周长和长、宽关系列方程；
   4. 解方程求出长和宽；
   5. 用长 × 宽算出面积。”
   ```


- **树状推理（Tree-of-Thought，ToT）**：将问题拆解为多分支的 “决策树”，每个节点代表一个推理步骤，模型在每个节点评估可能的路径，选择最优方向继续推导，类似人类 “试错 - 调整” 的思考模式。
   ```markdown
   场景：解决 “如何在 1 小时内提升某款奶茶店的线下销量”
    
   Prompt：
   “请帮我思考提升奶茶店 1 小时内销量的方法。步骤要求：
   1. 先列出 3 个可能的方向（如 “降价促销”“增加品类”“快速引流”）；
   2. 针对每个方向，分析具体做法（如 “降价促销” 可拆分为 “买一送一”“第二杯半价”“满减券”）；
   3. 评估每个做法的可行性（如 “买一送一” 是否会导致原料不足？“快速引流” 是否需要店员外出拉客？）；
   4. 排除明显不可行的选项，最终保留 2 个最优方案并说明理由。”
   ```
   
- **逐步拆解推理（Step-by-Step Decomposition）**：将复杂问题按 “总目标→子目标→具体步骤” 层层拆解，每个子目标对应可执行的小任务，最终通过完成子任务达成总目标。
  ```markdown
  场景：完成 “3 天内写一篇关于‘AI 对教育影响’的 5000 字论文”
  
  Prompt：
  “请帮我拆解写论文的步骤。要求：
  1. 总目标：3 天内完成 5000 字论文，包含摘要、引言、3 个论点、结论。
  2. 按天拆解子目标：
  1. 第1天需要完成什么？（如确定论点、收集文献）
  2. 第2天需要完成什么？（如撰写初稿、补充案例）
  3. 第3天需要完成什么？（如修改逻辑、调整格式）
  ```

- **反向推理（Backward Reasoning）**：将复杂问题按 “总目标→子目标→具体步骤” 层层拆解，每个子目标对应可执行的小任务，最终通过完成子任务达成总目标。
  ```markdown
  场景：解决 “如何在 2 小时内让一个零基础者学会包饺子”

  Prompt：
  “请从‘2 小时后学会包饺子’这个结果倒推步骤：
  1. 学会包饺子的最终标志是‘能独立包出 10 个合格饺子’，为此需要具备哪些能力？（如 “会擀皮”“会调馅”“会捏褶”）
  2. 每个能力需要什么前提？（如 “会擀皮” 需要 “知道面团如何揉制”“擀面杖使用方法”）
  3. 从零基础开始，按逆序排列学习步骤（如 “先学调馅→再学擀皮→最后学捏褶”），确保 2 小时内可完成。”
  ```

- **证据链推理（Evidence Chain Reasoning）**：围绕结论收集多维度证据，通过 “证据→关联→结论” 的链条验证合理性，强调每个结论必须有明确证据支撑，且证据之间无矛盾。
  ```markdown
  场景：分析 “某公司是否存在财务造假”

  Prompt：
  “请判断某公司是否财务造假，要求：
  1. 列出 3 个关键证据维度（如 “利润表异常”“现金流与营收不匹配”“审计报告疑点”）；
  2. 每个维度需提供具体证据（如 “利润表中连续 3 年毛利率高于行业均值 50%”“经营现金流为负但净利润为正”）；
  3. 说明证据之间的关联性（如 “高毛利率是否依赖虚构收入？现金流为负是否因收入未实际到账？”）；
  4. 若证据形成闭环（无矛盾、能相互印证），再下结论。”
  ```

- **类比推理（Analogical Reasoning）**：通过 “已知场景 A” 与 “未知场景 B” 的相似性，将 A 的推理逻辑迁移到 B 中，即 “因为 A 和 B 在 X 方面相似，所以 A 的结论可适用于 B”。

| 推理模型 | 核心特点    | 适用场景 |
|----|----|----|
| 非结构化推理 | 无固定步骤，依赖直觉 | 简单主观问题（如日常选择）|
| 思维链 | 线性分步推导，展示推理过程 | 数学解题、逻辑问答等需清晰推导路径的场景 |
| 树状推理 | 多分支试错，选最优路径 | 创意生成、复杂谜题 |
| 步拆解推理 | 层级化拆分任务 | 项目管理、流程规划 |
| 反向推理 | 从结果倒推条件 | 定理证明、目标达成规划 |
| 证据链推理 | 依赖多维度证据闭环 | 法律分析、科学论证 |
| 类比推理 |  跨场景相似性迁移 | 创新设计、跨界方案 |

## OpenAI
### 关键流程
1. **用户交互层（Input Handling）** 
   1. 多模态输入处理 
      1. 语音输入：通过 ASR（自动语音识别）将语音转为文本（如 “小爱同学，今天天气如何？”） 
      2. 文本输入：直接接收用户文字信息 
      3. 其他输入：图片、手势（需特定场景支持） 
   2. 输入规范化
      1. 纠错：修正拼写错误（如 “请帮我查下明天的天气” → “请帮我查下明天的天气”）
      2. 标准化：统一表达形式（如 “后天” → “2025 年 7 月 23 日”）
2. **自然语言理解（NLU：Natural Language Understanding）**
   1. 意图分类
      1. 判断用户意图类型（如查询天气、订购外卖、闲聊）
      2. 例：“明天北京会下雨吗？” → 天气查询意图
   2. 实体提取
      1. 识别关键实体（时间、地点、物品等）
      2. 例：从上述问题中提取 “明天”（时间）、“北京”（地点）
      3. 将提取的实体填充到预定义模板（槽位）中 {"意图": "天气查询","槽位": {"时间": "2025-07-22","地点": "北京"}}
   3. 对话状态跟踪
      1. 维护多轮对话的上下文（如用户追问 “那后天呢？” 需关联上一轮的 “北京”）
3. **对话管理（DM：Dialogue Management）**
   1. 根据当前状态选择回复策略（如直接回答、反问澄清、调用外部 API）
       1. 如果 意图 == "天气查询" and 槽位完整:return "调用天气API获取数据"else:return "询问缺失槽位（如‘您想查询哪个城市的天气？’）"
   2. 知识调用
       1. 从知识库 / API 获取回答所需信息，例：调用 OpenWeather API 获取北京 2025-07-22 的天气预报
4. **自然语言生成（NLG：Natural Language Generation）**
   1. 模板生成
      1. 基于预定义模板填充内容（早期方法）
      2. 例：“{地点} 明天 {天气状况}，气温 {温度范围}” → “北京明天多云，气温 25-30℃”
   2. 模型生成
      1. 使用 LLM（大语言模型）生成自然流畅的回复
      2. 例：“根据中国气象局的预报，北京明天（7 月 22 日）以多云天气为主，气温 25-30℃，非常适合户外活动～”
   3. 风格适配
      - 根据用户画像调整回复风格（如对儿童使用更简单的语言）
5. **输出呈现（Output Generation）**
   1. TTS（文本转语音）
      - 将生成的文本转换为语音（如 Siri 的语音合成）
   2. 多模态呈现
      - 搭配图片、视频等输出（如天气查询结果附带雷达图）
      -  回复发送
   
### 代码示例
> 对话API文档 : https://platform.openai.com/docs/api-reference/chat/create

```python
import os
from openai import OpenAI


# 设置 OPENAI_API_KEY 环境变量
os.environ["OPENAI_API_KEY"] = "sk-xxxx"
# 设置 OPENAI_BASE_URL 环境变量
os.environ["OPENAI_BASE_URL"] = "https://api.openai-proxy.org/v1"

# 初始化 OpenAI 客户端
client = OpenAI(
    api_key=os.environ.get("OPENAI_API_KEY"),
    base_url=os.environ.get("OPENAI_BASE_URL"),
)


# 任务描述
instruction = """
你是一名专业的国宴大厨陈师傅，任务是根据用户输入的食材，推荐合适的菜谱。
这是一个菜谱推荐系统，给出详细的菜谱步骤，包括所需的调料、烹饪时间和火候。
"""

# 输出描述
output_format = """
给出的菜谱步骤要清晰易懂，调料用量要适中，烹饪时间和火候要合理。
语气要友好、专业，让用户能轻松上手操作。
"""

# 用户输入
input_text = """
{
  "ingredients": ["鸡肉", "土豆", "青椒"],
  "dietary_preference": "无辣不欢"
}
"""

# prompt 模版
prompt = f"""
{instruction}

{output_format}

用户输入：
{input_text}
"""

def openai_chat(prompt_text: str):
    completion = client.chat.completions.create(
        model="gpt-3.5-turbo",
        temperature=0.0, # 控制生成结果的多样性 0.0 表示完全确定，值越大越随机
        messages=[{"role": "user", "content": prompt_text}],  # type: ignore # 要生成稳定的状态数据 , 设置最小随机性
    )

    print(completion.choices[0].message.content)


def main():
    openai_chat(prompt)


if __name__ == "__main__":
    main()
```

- **核心参数**
  - **temperature** : 控制生成结果的多样性 ; 默认值为 0.7 , 取值范围 0 ~ 2。
    - 值越高 , 生成的结果越随机 , 设置为 2 基本胡说八道 ;
    - 值越低 , 生成的结果越固定 , 0 的时候基本固定 ;
  - **seed** : 随机种子 , 如果不指定 则 OpenAI 自己随机决定用什么随机种子 。
    - 指定该 seed 参数后 , 如果 temperature = 0 , 则固定的种子生成的结果是固定的 ;
  - **stream** : 控制是否以数据流模式进行生成 ; 默认值为 False 。
    - 如果设置为 True , 在数据流模式下 , 一个字一个字的输出 , 浪费流量 ;
    - 默认 False , 整个生成完了 , 在一次性返回 ;
  - **top_p** : 在随机采样时 , 只考虑累计概率前百分之多少的 token , 有助于控制生成文本的多样性。
    - 与 temperature 参数作用相似 , 不建议与 temperature 一起使用 ;
  - **n** : 一次生成的结果数量。 
    - 使用提示词的 自洽性 时使用 , 一次返回多个结果再进行比较 ;
  - **max_tokens** : 每条生成结果的最大 token 数量 , 超过这个限制的部分会被截断。
    - 省钱方案 , 防止某个用户大量消耗 token ;
  - **presence_penalty** : 对已经出现过的 token 的概率进行降权 , 帮助生成更多样化的文本结果。
    - 避免输出很多重复的话 , 浪费 token ;
  - **frequency_penalty** : 根据 token 的出现频次 , 对其生成概率进行降权。
  - **logit_bias** : 对指定的 token 进行手工加权或降权 , 可以通过这个参数来调整特定 token 的生成概率 , 但不常用。
